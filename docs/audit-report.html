Executive Summary

Fox Passport Republic is a marketplace and event platform for the Philippines that connects Citizens (users), Foxers (event/service creators), Mayors (venue providers), and Administrators. The frontend is built on Next.js 16 with the App Router, TypeScript, TanStack Query, Zustand, and Tailwind CSS v4.

The codebase represents a mature MVP. Core booking, authentication, and administrative flows are implemented. Several secondary features (host analytics, KYC verification, notifications) are structurally present but incomplete. The build passes with no TypeScript errors.

Architecture Overview

Framework

  • Next.js 16 (App Router)
  • React 19 (Server + Client Components)
  • TypeScript — strict mode
  • Middleware for auth routing

State Management

  • TanStack Query v5 — server state
  • Zustand — UI & form state (11 stores)
  • SessionStorage — checkout persistence
  • Cookies — SSR auth tokens

Styling

  • Tailwind CSS v4
  • shadcn/ui + Radix UI primitives
  • Lucide React & Material Symbols icons
  • Motion (Framer Motion) animations

API & Integrations

  • Axios — REST client (base: /api/v1)
  • Stripe — payment processing
  • Supabase — additional backend features
  • Mapbox GL — venue maps

Validation & Schema

  • Zod — 13 schemas defined
  • React Hook Form — form management
  • Server-side auth guards (requireAuth, requireAdmin, requireHost)

Auth Strategy

  • JWT access + refresh tokens
  • localStorage + HTTP-only cookies
  • Middleware route protection
  • Silent refresh on 401
  • 5 user roles (citizen, host, mayor, foxer, admin)

Data Flow Pattern

Server Components fetch initial data via lib/server/data.ts (with auth headers from cookies) and pass it as props to Client Components. Client Components use TanStack Query hooks for subsequent mutations and refreshes. Zustand stores hold UI-only state and builder form state.

5.3 — UI Consistency Issues

  • Dual icon systems: Both Lucide React and Material Symbols are used side-by-side. This adds bundle weight and creates visual inconsistency between icon weights and styles.
  • Inline any types in props: Several admin table components accept any[] props instead of typed interfaces. This bypasses TypeScript's safety in UI rendering.
  • No loading skeletons on most pages: Only spinner fallbacks are used; skeleton screens would improve perceived performance significantly.
  • Empty state screens: The host dashboard uses a generic EmptyState component, but user dashboard and foxer profile pages lack proper empty states.
  • Form error messages: Inconsistent — some forms use toast notifications, others use inline field errors, and some have no error feedback.

Data Layer & API Integration

6.1 — API Client Structure

The client uses Axios with a configured base URL (NEXT_PUBLIC_API_URL, default http://localhost:3002/api/v1). A request interceptor adds the Bearer token; a response interceptor handles 401 with silent refresh. Server-side requests use a separate fetch-based helper in lib/server/data.ts that reads the token from cookies.

6.2 — Known Data Issues

  • Host dashboard mock data: useHostDashboard.ts computes stats from real venue data but hardcodes occupancy rates, revenue, and booking counts.
  • Admin charts empty data: AdminChartsSection likely receives empty arrays; charts may render blank axes.
  • User journey static: UserJourney component uses a hardcoded timeline structure, not real booking/activity history from API.
  • Polling instead of invalidation: useAdminData and useHostData use a polling interval; TanStack Query's invalidateQueries (already used in admin tables) is the correct pattern.

Prioritized Recommendations

Critical — Fix Before Launch High user impact, data integrity, or security risk
Effort: 1–2 days

C-1 · Fix role field mismatch (user.role vs user.roleType[])

BookingConfigurationClient and other components read user?.role (singular), but the User type defines roleType: string[]. Multi-role users (e.g., someone who is both a host and a mayor) will get incorrect dashboard redirects. Standardize to a single field across all components and the auth store.

authtypesrouting
Effort: 1–2 days

C-2 · Wire real data to Host Dashboard KPIs

Host dashboard KPI cards (revenue, total bookings, occupancy rate) display hardcoded or computed-from-stub values. When hosts see incorrect numbers, trust is destroyed immediately. Connect useHostDashboard to real API endpoints for stats.

hostdataanalytics
Effort: 1 day

C-3 · Consolidate duplicate route trees (/host/* and /creator-dashboard/*)

Both route trees serve identical content. This creates split navigation targets, double maintenance, and SEO issues. Decide on one canonical path (recommend /creator-dashboard) and redirect the other. Remove duplicated components and page files.

routingarchitecture
Effort: 2–3 days

C-4 · Build "Forgot Password" flow

The auth modal has no password reset option. A production marketplace cannot launch without this. Implement forgot-password request and reset-password confirmation routes.

authUX
Effort: 2–3 days

C-5 · Implement booking cancellation / refund flow

Citizens currently have no way to cancel a booking they made. This is a legal and trust requirement for any marketplace. Add a cancellation page accessible from the user dashboard bookings list with configurable cancellation policy display.

bookingsUXlegal
High — Address Before Public Beta Significant UX gaps or features needed for core flows
Effort: 2–3 days

H-1 · Replace polling with TanStack Query invalidation

useAdminData and useHostData use interval polling (e.g., every 30s) to refresh data. This burns bandwidth and creates stale data windows. Replace with queryClient.invalidateQueries() after mutations — which is already done correctly in the admin approve/reject tables.

performancedata-fetching
Effort: 2–4 days

H-2 · Complete the Foxer Profile page

/foxer/[id] is a key discovery surface but renders an empty shell. Build out the portfolio section, list the foxer's published event templates, show ratings/reviews, and add a "Book This Foxer" CTA that leads to the matching flow.

H-3 · Add error boundaries to all Client pages

No Client Component pages have React error boundaries. A single failed API call or undefined access will crash the entire page with a generic "Something went wrong" screen. Add page-level error boundaries using Next.js error.tsx per-segment files with recovery UI.
